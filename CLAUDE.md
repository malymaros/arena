# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A 2-player turn-based browser arena game (wizard duel on a 5×3 grid) built with Express + Socket.IO. No build step, no tests, no linter — plain JS served statically. Code comments are in Slovak.

> **Adding a new character?** Read `docs/ADDING_CHARACTERS.md` first — a step-by-step guide (files to touch, questions to ask up front, and the edge-case matrix: defenses, labyrinth, Last Stand, tournament, clone/summon). It exists because adding Naruto touched far more systems than expected.

## Commands

```
npm start        # node server.js → http://localhost:3000
npm test         # integration tests — boots the server on :3996, drives 2 socket clients (test/game-test.mjs)
```

Environment variables: `PORT` (default 3000), `ADMIN_KEY` (optional password for admin reset; if unset, reset is open), `FORCE_FIRST_STARTER` (`A`|`B` — pins the otherwise random game-1 starter; used by the tests).

## Architecture

Two files contain all the logic:

- `server.js` — authoritative game state and turn resolution (single global game; two **persons** A/B map to board **slots** `p1`/`p2`)
- `public/client.js` — rendering, animation, and input; trusts the server's state but replays it via timelines

### Match lifecycle, lobby & series

`game.phase` is a state machine: `lobby → (team_select) → playing → match_over` (`team_select` only in the tournament format). The **first connected person (A) is the host** and sees the lobby setup screen; person B sees a waiting screen. The host emits `configure_match` (`{ format, tilesPerRound, tileWeights, timer }`, validated/sanitized server-side; tile percentages must sum to exactly 100) which starts the match. Format is `single | bo3 | tournament` (`MATCH_FORMATS` = wins needed: 1/2/3). Identity is the **person** (A/B); the **slot** `p1`/`p2` is the left/right board role. At `configure_match`, `startMatch()` **randomly assigns persons to slots** (`game.seats`; `FORCE_FIRST_STARTER=A|B` env pins who gets `p1` — the tests rely on this) and the seats are then **fixed for the whole series**. `p1` = **white**, left, and **always starts game 1**; in a series (bo3/tournament) the game starter **alternates per game** (odd games open with `p1`/white, even with `p2`/black); within a game the per-round starter flips each round (turn resets to 1 each game). After the roll the server broadcasts `color_roll`; the client then plays a **spinning-arrow yin-yang animation** (`#color-roll` overlay above char-select) whose pointer lands on the player's own colour — the assignment already happened server-side, the wheel is pure theatre. The right side (`p2`, black) always renders its mages with the alternative "albino" palette (`alt-color`), consistently across board, HUD portrait, ghost, projectiles, tournament mage heads and char-select; `p2`'s char-select is additionally **mirrored** (`#char-select.p2-side` flips card sprites and the mini-caster, `renderAbilityPreview` mirrors the caster position and the directional special's direction). Series score is tracked **per person** in `game.seriesWins` and exposed per current slot via `seriesSnapshot()`. Characters are **re-picked before every game** (each game clears `char` → char-select reappears). When a game ends, `handleGameEnd()` records the win and emits `game_result` (`{ gameWinner, series, matchOver }`); on `matchOver` it also emits `game_over` (series decided), otherwise it schedules the next game (`new_game` + fresh `playing` state) after the client has played out the timeline. The **turn timer** (`config.timer` = `off | 10 | 30 | 60 | quickdraw`) is **client-driven**: the client counts down and on expiry auto-fills the 3 basic action slots with random valid actions and auto-locks (golden actions are never auto-added); the server just validates the resulting `lock_in`.

### Tournament (team draft + HP/mana carry-over)

First-to-3 series where each person drafts a **team of `TEAM_SIZE` (3) characters** from the full pool of 5. After `configure_match` the phase goes to `team_select`: each player emits `choose_team` (3 unique characters, once per match). The draft is **blind** — `snapshotFor` masks the opponent's `roster` until both confirmed; `rosterReady` stays public so the client can show "opponent is ready". When both confirm, `finishTeamSelect()` builds `game.mageHp`/`game.mageMana` (keyed per person by their roster) and starts game 1. Every mage's HP and mana **carry over between games** (saved at game end and at swap); a defeated mage drops to 0 HP = permanently dead — cannot be picked or swapped in (skull head in the HUD, dead card in char-select). Characters are re-picked before each game **from the player's own roster only** (client `roster-mode` merges both char-select pages via `display: contents` and hides off-roster cards). During a round a player may queue up to **2 `swap` actions** (`{type:"swap", to}` — free, two-phase teleport animation, position stays; target must be an own-roster living mage, not the mage the round started with, each target at most once) — validated in `validQueue()`, executed in `doSwap()`; the rest of the round is played by the new mage. The HUD shows each side's roster as **3 animated mage heads** (`renderMageHeads`, order = draft order from `state.roster`); own living non-current heads toggle a swap on click. `rosterHp`/`rosterMana` (stored per-mage values per slot) are public. **Tournament × labyrinth:** while any labyrinth is active, swap is **forbidden for both sides** (rejected at `lock_in`; a swap already queued when the curse lands mid-round resolves as `invalid`) and the mage heads are hidden on both sides (`.lab-hidden`, reappearing at `labyrinth_end`); the opponent's `rosterMana` is redacted **both ways** (like live mana); a **lethal tile/IK hit** during a labyrinth plays the same reveal sequence as an action hit (`revealLabyrinths` before the tile frames, `endLabyrinths` after the hit) so a game never ends inside the fog. **Tournament × Last Stand:** the buffed player cannot swap in the final round (rejected in `validQueue`, guarded in `doSwap`, heads not clickable) — otherwise the demon's full heal would be banked into the roster at swap time and the doom would kill the substitute instead of the mage who made the pact.

### Server-authoritative timeline model

The core pattern: the client does **not** simulate the game state. Each turn, both players queue exactly 3 actions (`move`, `dash`, `recharge`, `attack`, `melee`, `special`, `shield`, `mirror`) — each action type at most once per round, enforced both in the client UI and in server-side `validQueue()` — and emit `lock_in`. Two optional extra actions can wrap the queue: the round's **non-starter** may prepend a **golden pre-action** — either `golden_shield` (3 mana, behaves exactly like a shield) or `golden_mirror` (5 mana, behaves exactly like a mirror), the two are mutually exclusive and both resolve before the starter's first action — and **either player** may append `golden_mana` (resolved after everything else in the round; +6 mana for HP, the HP cost grows by 1 with each use per game — tracked in `manaRefills`, refused if it would kill or mana is full). `move`, `dash` and `attack` carry a `dir` (up/down/left/right): the basic attack fires in the chosen direction and hits the first opponent in its path (damage falloff 3/2/1 by distance, never its own cell), so aiming is a prediction of where the opponent will be when the action resolves. `dash` (4 mana) moves up to 2 cells in one direction (clamped at the board edge). `melee` (4 mana, 8 dmg) hits only an opponent sharing the caster's cell. **Action into a wall:** a directional action that "has nowhere to go" (move/dash into the board edge, or an attack fired off the board from an edge cell) is **not crossed out and is not free** — if you have the mana it is spent and the action *executes* in place (dash/move → `wall_bump` OUT-OF-BOUNDS twitch; attack → an `attack_swing` whiff frame, the shot fizzling at the edge). This closes the old exploit where a player filled slots with impossible-into-wall actions to passively hoard mana. Only a genuine **lack of mana** still yields an `invalid` (crossed-out, refunded); `move` is free so it always just executes. For planning previews the client simulates its **own** queued moves ("ghost"): a translucent sprite shows the post-move position and aim/special previews are computed from the position valid at that point in the queue (`simulatedPositions()`/`ghostPos()` in `client.js`). When both are locked, `resolveTurn()` executes actions interleaved (the round starter flips each round; game 1 is always opened by `p1`/white, in a series the game starter alternates per game), and builds a **timeline** — an array of state snapshots, each with an `effects` array (`charge`, `hit`, `recharge`, `special`, `invalid`) and a `delayMs`. The whole timeline is emitted in one `state` event; the client plays it back frame-by-frame with `setTimeout` in `schedulePlayTimeline()`, mapping effects to sprite animations.

When changing game logic, the server-side `do*` action functions must push frames via `pushStateFrame()` for anything the client should animate — state changes without frames render as teleports.

Turn resolution checks for a winner (`hp <= 0`) after **every individual action and every individual tile hit** and aborts the rest of the turn on lethal — the first death ends the game, so a draw cannot occur. At game end `game_result` is emitted alongside the final timeline (and `game_over` too if the series is decided); the client deliberately delays the overlay until attack/death animations finish (`serverWinner`/`serverGameResult` + `showGameOverSequence`, or `playGameEndAnim` → `showIntermission` between games in a series).

### Socket protocol

Client → server: `configure_match` (host only, in `lobby` phase), `choose_team` (tournament only — 3 unique characters, in `team_select` phase, once per match), `choose_character` (fire | lightning | wanderer | medusa | minotaur | naruto — only in `playing` phase, once per game; in tournament only from the own roster), `lock_in` (array of 3 actions, or 4 with `golden_shield`/`golden_mirror` first — only valid from the round's non-starter; only in `playing` phase; in tournament the queue may contain up to 2 `swap` actions), `retry`, `admin_reset_all`.
Server → client: `you_are` (`{ slot, isHost }`; slot is rolled at match start and fixed for the series), `color_roll` (after `configure_match` — trigger for the client-side colour-wheel animation; the seat assignment is already done), `state` (snapshot incl. `phase`/`config`/`series`, tournament also `roster`/`rosterReady`/`mageDead`/`rosterHp`/`rosterMana` + own `mageHp`/`mageMana`, optionally with `timeline`), `game_result` (per-game outcome), `new_game` (next game in series — reset round UI, keep score), `game_over` (series decided), `reset`.

### Game balance constants

All tuning lives at the top of `server.js`: HP/mana, costs, damage, and animation pacing (`*_MS` constants — currently 2× slowed). Client-side timing constants in `client.js` (`MOVE_MS`, `ATTACK_SWING_MS`, etc.) must stay in sync with the server's `delayMs` values or animations desync from the timeline playback.

Character specials differ by hit zone: fire hits the whole row (5 dmg), lightning hits every cell of the **opposite chess-color** to the one it stands on — half the board, and each orthogonal move flips its parity (3 dmg) — wanderer hits diagonal-adjacent only (8 dmg). Medusa's special is directional (left/right), deals no damage and **petrifies** (target skips its next 2 basic actions; golden actions are skipped without consuming the counter). Minotaur's special covers the **whole board**, deals no damage and banishes the opponent into the **labyrinth** (see below). Naruto's special is **self-range** and summons a **shadow clone** (see below). The hit logic is in `specialDamageAndHit()` / the medusa+minotaur+naruto branches of `doSpecial()` (server) and must match `cellsForSpecialPreview()` (client hover/cast preview) — they are maintained in parallel.

**Shadow clone (Naruto):** the special (5 mana, cannot be blocked/mirrored) requires Naruto to stand on his cell **without the opponent** — the opponent standing on Naruto's cell is the **only way to block the summon**, so it still **costs the mana** (any existing clone is killed first — as in a recast — then seals play + a `not_alone` whiff, but no new clone spawns); a recast first kills the active clone (as if hit), then summons a fresh one on Naruto's cell. State lives in `player.clone = {x,y} | null` (serialized in snapshots; `redactActor` strips it for a cursed viewer). The clone **copies every basic action in parallel** with its owner — same frames/beats — with one inversion: **the vertical axis is mirrored for both movement and the basic attack** (owner moves up → clone moves down; owner shoots up → clone shoots down; horizontal is identical; dash keeps its own board clamp via `moveCloneSteps`). A vertical move/dash/attack at the board edge makes whichever figure is blocked play a **`wall_bump`** (OUT OF BOUNDS twitch) while the other still moves and mana is still spent; when *neither* figure can move the action still **executes** (both play `wall_bump`, mana still spent — it is never crossed out, see the "action into a wall" rule below). Because both figures split symmetrically on a vertical action, the opponent's **action log / round-script anonymizes Naruto's vertical direction to `↕`** (`displayDir` — up/down would reveal which figure is real; horizontal stays). Copied attacks/melee deal the **same damage as Naruto himself** — the basic shot uses full distance falloff (`Math.max(1, BASIC_DMG_MAX - dist)`) measured from the clone's own cell, melee deals the full `MELEE_DMG`, both scaled by the owner's Last Stand/Last Hope (`dealMul` ×2 / ×4) and maze (`labyrinthMul` ×2) multipliers. Because a clone **stacked on the owner's cell** fires/melees the same target in parallel, the pair lands two full hits = **double damage** (this stacks only for horizontal shots and melee; a vertical shot still splits up/down via the inversion). `CLONE_DMG = 1` now denotes only the clone's **soak** — the flat amount it eats when a shot passes through the stacked pair before the rest carries on to Naruto (its offense and its mirror reflect are both full, Naruto-equal); recharge is visual-only on the clone (mana is the owner's single pool — the `recharge` effect carries both cells so the client shows the aura on both); defenses are the **owner's shared `shield`/`mirror` flags** (armed and consumed together, identical glow on both sprites — the pair is visually indistinguishable; the client duplicates defense/mana floats via `cloneFloat`). The clone has no HP: **any connecting hit removes it** (`killClone` → `clone_die` frame) — enemy projectiles (a clone standing **apart** from the owner is a decoy that eats a shot whole and lets it fly on as smoke; a clone **stacked on the owner's cell** soaks only `CLONE_DMG` and the **rest of the shot passes through to the owner** — it is not a full shield, and in a labyrinth that surplus hit on the real owner reveals & ends the maze like any other), melee, zone specials (they strike player **and** clone in the same action; a shared defense reacts on **both figures at once, each from its own cell, in a single synchronized beat** — the zone special and the demon route through `applyHitBoth(ownerSlot, ownerDmg, tl, kind, includeClone)`, which emits both blocks / both mirror-beams / both hits in the *same* frame so nothing plays sequentially: a **shield** breaks on the player *and* the clone (no damage either way), and a **mirror** reflects from both — both bounce the **full incoming dmg** (the clone reflects the same damage it took, exactly like the player's mirror — not a flat `CLONE_DMG`), so a mirrored wanderer special that hit both deals the attacker `8 + 8` simultaneously; undefended, the player takes the dmg and the clone poofs in the same beat. `applyHitOnClone` (used for clone-only hits and the stacked-shot/melee paths, which stay sequential to preserve the attacker-clone `fromClone` reflect + partial-absorb) reflects the **full incoming dmg** back at the attacker), Medusa's petrify and Minotaur's banish (status specials destroy the clone unless defended; a clone-side mirror reflects the status onto the caster), the demon attack, and **IK tiles**. A dmg tile shows a cosmetic `clone_hit` −1 float but does not kill it; the clone never consumes heal/mana pickups (an untaken pickup betrays which figure is the clone). An opponent's mirror reflecting a **clone-originated** hit destroys the clone instead of damaging the owner (`applyHit(..., fromClone)`). The clone dies with the owner's death, swap (`doSwap`) and at game end (`resolveTurn` `ended` cleanup). **Clone × labyrinth:** a hit on a **decoy** clone (standing apart from the owner) never reveals nor ends a labyrinth and the cursed shooter must not learn about it — the killing projectile flies on to the board edge (smoke pass-through); a hit on a **stacked** clone is different, since the surplus damage lands on the real owner and that hit reveals & ends the maze normally and `clone_die`/`clone_hit` effects are redacted like other opponent-side effects; a hit **by** the clone is a normal action hit (ends the labyrinth, blocked/mirrored included). The clone counts for Ariadne-thread crossings both ways (`trackCloneSteps` + the clone check in `trackSteps` — the cursed player sees a silhouette at the meeting cell, not knowing it was the clone). Client-side the clone renders via per-slot `cloneEls` canvases (positioned in `positionActors`, drawn in the raf mirroring the owner's `animState` except hurt/dead — a tile hit on the clone triggers an independent `cloneFlinch` hurt-pose + `-N HP` float so it doesn't read as invulnerable; it does **not** mirror Last Stand visuals — `actorFilter(slot, now, true)` skips the golden death-glow. The clone is **always visible** (both sides — you can always see Naruto has a clone): while sharing the owner's cell it stands a few px beside him (`STACK_OFFSET`) and semi-transparent, and when separated it is full-opacity and identical; it **always slides** (never pops/vanishes in place), so splitting and merging read as two independent figures moving in parallel from the shared cell) plus a `cloneGhost` planning preview (`simulatedClonePos`); the summon choreography is `clone_summon` (owner + one copy on the same cell, mirrored to face the owner, both play `Special_2.png` via `spawnCloneSummonFx`) followed by `clone_born`.

**Labyrinth (Minotaur):** the cursed player sees only the cell they stand on **plus Damage tiles and all IKs** (`redactTilesFor`: dmg tiles + `iks` are always visible so the blind player can dodge them; **heal/mana tiles are redacted unless the cursed player stands directly on that cell**) — the server **redacts their data** (`snapshotFor` + `redactTimelineFor` in `emitStateMasked()`): opponent position/defenses/**mana** and all opponent-originated timeline effects are stripped (incl. `tile_proc` blinks outside the player's own cell); opponent `action` announcements are masked as `{type:"unknown", beat}` so the round-script bar shows "?" on the right beat. Redaction is **two-way**: the hunter (Minotaur) does *not* see the Ariadne's thread, the `threadMark`, `thread_mark` effects, nor the cursed player's **mana** (`redactHunterActor`) — during the labyrinth neither side can track the opponent's mana (HP stays in the data but is visually hidden too); on **both** sides the opponent's HUD widget is **covered** by `assets/minotaur/labyrint.png` (`.hud-box.in-maze .maze-cover` — portrait, HP, mana, action log and the YOU/OPPONENT tag are hidden; the starter flag and series crowns stay visible). While anyone is in the labyrinth the arena background dims for **both** players (`body.labyrinth-mode`); the cell fog applies only to the cursed one. The effect lasts until **either player lands an action hit on the other — a blocked or mirrored hit counts too** (`endLabyrinths()` is called in `applyHit`/`applyPetrify`/`applyLabyrinth` *after* the block/reflect/hit frames, so the hit animation and HP loss play out before the ESCAPED float and the arena un-dims; in `applyLabyrinth` it runs before the new curse lands so two labyrinths never run at once; tile damage does *not* end it). Exception — **deterministic deaths during a labyrinth**: a lethal tile/IK hit and the Last Stand doom banish play the full reveal sequence (`revealLabyrinths` before their frames, `endLabyrinths` after the death) so a game never ends inside the fog with redaction still active. Because a hit is deterministic at action start, an action that will land one triggers `revealLabyrinths()` **before its animation frames**: a `labyrinth_reveal` frame sets the actor flag `labReveal`, which drops the redaction, the fog and the widget cover immediately (the hidden opponent fades in client-side, demon-style; thread and `threadMark` die at reveal) while `labyrinth` itself stays true through the action — the arena stays dimmed until the real `labyrinth_end` lands after the hit. While cursed, every cell the player enters (dash counts intermediate cells) extends their **Ariadne's thread** (`player.thread`, starts on the cell where they were cursed); `player.threadMark` records the **last cell where the thread and the Minotaur met** — whether the Minotaur stepped onto the thread *or* the cursed player walked onto the Minotaur's cell — and the cursed player sees the opponent's silhouette there. **Hunter on the cursed player's own cell:** the cursed player's own cell is always visible ("torch-lit", `.cell.torch-lit`), so when the hunter (or its shadow clone) currently shares that cell the server exposes a single boolean `hunterHere` on the cursed player's own snapshot actor (`redactSnapshotFor` — the only positional info leaked, and recomputed each snapshot so it clears the instant the hunter leaves; the hunter's real `x/y` stay redacted). The client then draws the silhouette **lit** (real palette, not the black `brightness(0)` shadow — `.sprite-silhouette.lit`) dodged aside on that cell, so the player sees the Minotaur as if outside the maze; otherwise the silhouette stays the black `threadMark` breadcrumb. Thread and mark die with the labyrinth. The special goes through defenses like petrify: shield blocks it, mirror reflects it — the Minotaur himself ends up in the labyrinth. Casting it on an already-cursed target is invalid (`already_lost`). **Balance — maze buff (`player.mazeBuff`):** a **successful direct cast** (curse actually lands on the foe — not blocked, not mirror-reflected) grants the caster `mazeBuff` for the labyrinth's duration (set in `applyLabyrinth`'s direct branch, cleared in `endLabyrinths`). It gives the caster **2× outgoing damage** (`labyrinthMul` multiplies the raw in `doBasic`/`doMelee` — since it's *outgoing*, a mirror reflects the **doubled** damage back onto the Minotaur); the caster takes **Damage & IK tiles normally** (as if outside the labyrinth — `endOfStepTileEffects` no longer skips them). It is granted only to the player who *cast* it, **regardless of character** — a mirror-created labyrinth grants **no** buff to anyone (the mirrorer never cast it; the reflected Minotaur is the cursed one). This makes the buff exclusive to the player actually playing the hunter Minotaur. Client-side the fog/thread/silhouette are rendered in `renderGrid()` (cells `.fogged`/`.thread-cell`, `renderThreadLines()`) and the raf (silhouette canvas + `.in-maze` widget cover); a hidden opponent has `x: null` — rendering paths guard on it (incl. `pairShift`, so a shared cell never leaks to the cursed player).

Defenses cover **the opponent's next action** after activation and are consumed by it even if it dealt no damage: shield (2 mana) blocks all damage, mirror (4 mana) reflects the full damage back at the attacker (applied raw — reflected damage cannot be shielded or re-reflected), golden shield (3 mana, pre-round extra action of the non-starter) sets the same `shield` flag, and golden mirror (5 mana, the same pre-round slot) sets the same `mirror` flag. The golden pre-action variants carry a `shieldGold`/`mirrorGold` flag purely for gold-tinted visuals (the `block`/`mirror` timeline effects carry a `gold` boolean). All are armed/consumed in the `resolveTurn()` loop, applied in `applyHit()`, and expire at end of round — they do **not** carry over. Strict interleaving means two own defenses are never armed simultaneously — each one covers exactly the opponent action that follows its activation.

Special tiles spawn at the end of every round — **`config.tilesPerRound` (1–3) rolls per round**, each typed by `config.tileWeights` (`dmg`/`heal`/`mana`/`ik` percentages summing to 100; `rollTileType()`). They resolve at the **end of each step** in `endOfStepTileEffects()`: dmg tiles are permanent (1 dmg if standing on them), heal/mana are the only consumables (taken by the round starter if both players share the cell), and IK tiles (10 dmg) are overlays that hide tiles beneath them. **IK is now a list (`game.iks`), multiple allowed** — every IK relocates each round (`relocateIKs()`) and may overlap other tiles and players; the only constraint is that two IK tiles never share a cell. Tile damage bypasses all defenses (shield/mirror).

HP and mana both cap at 10 and are rendered in the HUD as 10-segment bars (`renderBar()` in `client.js`). Tile size is rectangular, driven by the `--tile-w`/`--tile-h` CSS variables, which `client.js` reads at startup (`TILE_W`/`TILE_H`).

### Sprites and assets

Sprite sheets are horizontal strips in `public/assets/<char>/`; frame count is inferred from `width / height` (frames must be square). Animation definitions (file, fps, loop) live in `ANIM_DEF` / `SPECIAL_ANIMS` in `client.js`. Characters with a native right-side palette (medusa, minotaur, naruto) have `dirP2` in `CHAR_META` (two sprite folders) instead of the CSS `alt-color` filter; medusa/minotaur `Charge.png` projectiles are per-palette recolored copies of fire's fireball, naruto's is a chakra spiral (P2 gold). Minotaur's sheet only has Idle/Walk/Attack/Hurt/Dead — `SPRITE_FILE_ALIAS` in `client.js` maps the expected `Run.png`/`Attack_1.png`/`Attack_2.png` names onto them. Naruto's assets were extracted from a ripped sheet via the reproducible pipeline in `tools/sprites-naruto/` (see its README); `Special.png` = hand seals (special cast), `Special_2.png` = the clone-summon pose loop. Arena background layers are listed in the client-side `ARENAS_CLIENT` map and loaded from `public/arenas/<key>/`. Background layers render into low-res canvases (`ARENA_RES`, upscaled with `image-rendering: pixelated`).

### Admin reset

`GET /admin/reset-all?key=…` or the socket event `admin_reset_all` disconnects both players and recreates the game. The client shows a reset button when loaded with `?admin=1` (optionally `&key=…`).
