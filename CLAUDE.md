## Project

A 2-player turn-based browser arena game (wizard duel on a 5×3 grid) built with Express + Socket.IO. No build step, no tests, no linter — plain JS served statically. Code comments are in Slovak.

> **Adding a new character?** Read `docs/ADDING_CHARACTERS.md` first — a step-by-step guide (files to touch, questions to ask up front, and the edge-case matrix: defenses, labyrinth, Last Stand, tournament, clone/summon). It exists because adding Naruto touched far more systems than expected.

```
npm start        # node server.js → http://localhost:3000
npm test         # integration tests — boots the server on :3996, drives 2 socket clients (test/game-test.mjs)
```

## Architecture

Two files contain all the logic:

- `server.js` — authoritative game state and turn resolution (single global game; two **persons** A/B map to board **slots** `p1`/`p2`)
- `public/client.js` — rendering, animation, and input; trusts the server's state but replays it via timelines

### Match lifecycle, lobby & series

`game.phase` is a state machine: `lobby → (team_select) → playing → match_over` (`team_select` only in the tournament format). The **first connected person (A) is the host** and sees the lobby setup screen; person B sees a waiting screen. The host emits `configure_match` (`{ format, tilesPerRound, tileWeights, timer }`, validated/sanitized server-side; tile percentages must sum to exactly 100) which starts the match. Format is `single | bo3 | tournament` (`MATCH_FORMATS` = wins needed: 1/2/3). Identity is the **person** (A/B); the **slot** `p1`/`p2` is the left/right board role. At `configure_match`, `startMatch()` **randomly assigns persons to slots** (`game.seats`; `FORCE_FIRST_STARTER=A|B` env pins who gets `p1` — the tests rely on this) and the seats are then **fixed for the whole series**. `p1` = **white**, left, and **always starts game 1**; in a series (bo3/tournament) the game starter **alternates per game** (odd games open with `p1`/white, even with `p2`/black); within a game the per-round starter flips each round (turn resets to 1 each game). After the roll the server broadcasts `color_roll`; the client then plays a **spinning-arrow yin-yang animation** (`#color-roll` overlay above char-select) whose pointer lands on the player's own colour — the assignment already happened server-side, the wheel is pure theatre. The right side (`p2`, black) always renders its mages with the alternative "albino" palette (`alt-color`), consistently across board, HUD portrait, ghost, projectiles, tournament mage heads and char-select; `p2`'s char-select is additionally **mirrored** (`#char-select.p2-side` flips card sprites and the mini-caster, `renderAbilityPreview` mirrors the caster position and the directional special's direction). Series score is tracked **per person** in `game.seriesWins` and exposed per current slot via `seriesSnapshot()`. Characters are **re-picked before every game** (each game clears `char` → char-select reappears). When a game ends, `handleGameEnd()` records the win and emits `game_result` (`{ gameWinner, series, matchOver }`); on `matchOver` it also emits `game_over` (series decided), otherwise it schedules the next game (`new_game` + fresh `playing` state) after the client has played out the timeline. The **turn timer** (`config.timer` = `off | 10 | 30 | 60 | quickdraw`) is **client-driven**: the client counts down and on expiry auto-fills the 3 basic action slots with random valid actions and auto-locks (golden actions are never auto-added); the server just validates the resulting `lock_in`.

### Tournament (team draft + HP/mana carry-over)

First-to-3 series where each person drafts a **team of `TEAM_SIZE` (3) characters** from the full `CHARS` pool. After `configure_match` the phase goes to `team_select`: each player emits `choose_team` (3 unique characters, once per match). The draft is **blind** — `snapshotFor` masks the opponent's `roster` until both confirmed; `rosterReady` stays public so the client can show "opponent is ready". When both confirm, `finishTeamSelect()` builds `game.mageHp`/`game.mageMana` (keyed per person by their roster) and starts game 1. Every mage's HP and mana **carry over between games** (saved at game end and at swap); a defeated mage drops to 0 HP = permanently dead — cannot be picked or swapped in (skull head in the HUD, dead card in char-select). Characters are re-picked before each game **from the player's own roster only** (client `roster-mode` merges both char-select pages via `display: contents` and hides off-roster cards). During a round a player may queue up to **2 `swap` actions** (`{type:"swap", to}` — free, two-phase teleport animation, position stays; target must be an own-roster living mage, not the mage the round started with, each target at most once) — validated in `validQueue()`, executed in `doSwap()`; the rest of the round is played by the new mage. The HUD shows each side's roster as **3 animated mage heads** (`renderMageHeads`, order = draft order from `state.roster`); own living non-current heads toggle a swap on click. `rosterHp`/`rosterMana` (stored per-mage values per slot) are public. **Tournament × labyrinth:** while any labyrinth is active, swap is **forbidden for both sides** (rejected at `lock_in`; a swap already queued when the curse lands mid-round resolves as `invalid`) and the mage heads are hidden on both sides (`.lab-hidden`, reappearing at `labyrinth_end`); the opponent's `rosterMana` is redacted **both ways** (like live mana); a **lethal tile/IK hit** during a labyrinth plays the same reveal sequence as an action hit (`revealLabyrinths` before the tile frames, `endLabyrinths` after the hit) so a game never ends inside the fog. **Tournament × Last Stand:** the buffed player cannot swap in the final round (rejected in `validQueue`, guarded in `doSwap`, heads not clickable) — otherwise the demon's full heal would be banked into the roster at swap time and the doom would kill the substitute instead of the mage who made the pact.

### Server-authoritative timeline model

The core pattern: the client does **not** simulate the game state. Each turn, both players queue exactly 3 actions (`move`, `dash`, `recharge`, `attack`, `melee`, `special`, `shield`, `mirror`) — each action type at most once per round, enforced both in the client UI and in server-side `validQueue()` — and emit `lock_in`. Two optional extra actions can wrap the queue: the round's **non-starter** may prepend a **golden pre-action** — either `golden_shield` (3 mana, behaves exactly like a shield) or `golden_mirror` (5 mana, behaves exactly like a mirror), the two are mutually exclusive and both resolve before the starter's first action — and **either player** may append `golden_mana` (resolved after everything else in the round; +6 mana for HP, the HP cost grows by 1 with each use per game — tracked in `manaRefills`, refused if it would kill or mana is full). `move`, `dash` and `attack` carry a `dir` (up/down/left/right): the basic attack fires in the chosen direction and hits the first opponent in its path (damage falloff 3/2/1 by distance, never its own cell), so aiming is a prediction of where the opponent will be when the action resolves. `dash` (4 mana) moves up to 2 cells in one direction (clamped at the board edge). `melee` (4 mana, 8 dmg) hits only an opponent sharing the caster's cell. **Action into a wall:** a directional action that "has nowhere to go" (move/dash into the board edge, or an attack fired off the board from an edge cell) is **not crossed out and is not free** — if you have the mana it is spent and the action *executes* in place (dash/move → `wall_bump` OUT-OF-BOUNDS twitch; attack → an `attack_swing` whiff frame, the shot fizzling at the edge). This closes the old exploit where a player filled slots with impossible-into-wall actions to passively hoard mana. Only a genuine **lack of mana** still yields an `invalid` (crossed-out, refunded); `move` is free so it always just executes. For planning previews the client simulates its **own** queued moves ("ghost"): a translucent sprite shows the post-move position and aim/special previews are computed from the position valid at that point in the queue (`simulatedPositions()`/`ghostPos()` in `client.js`). When both are locked, `resolveTurn()` executes actions interleaved (the round starter flips each round; game 1 is always opened by `p1`/white, in a series the game starter alternates per game), and builds a **timeline** — an array of state snapshots, each with an `effects` array (`charge`, `hit`, `recharge`, `special`, `invalid`) and a `delayMs`. The whole timeline is emitted in one `state` event; the client plays it back frame-by-frame with `setTimeout` in `schedulePlayTimeline()`, mapping effects to sprite animations.

When changing game logic, the server-side `do*` action functions must push frames via `pushStateFrame()` for anything the client should animate — state changes without frames render as teleports.

Turn resolution checks for a winner (`hp <= 0`) after **every individual action and every individual tile hit** and aborts the rest of the turn on lethal — the first death ends the game, so a draw cannot occur. At game end `game_result` is emitted alongside the final timeline (and `game_over` too if the series is decided); the client deliberately delays the overlay until attack/death animations finish (`serverWinner`/`serverGameResult` + `showGameOverSequence`, or `playGameEndAnim` → `showIntermission` between games in a series).

### Socket protocol

Client → server: `configure_match` (host only, in `lobby` phase), `choose_team` (tournament only — 3 unique characters, in `team_select` phase, once per match), `choose_character` (fire | lightning | wanderer | medusa | minotaur | naruto | escanor | soldier | werewolf, plus the side-bound countess (p1 only) / onre (p2 only) outside tournament — only in `playing` phase, once per game; in tournament only from the own roster), `lock_in` (array of 3 actions, or 4 with `golden_shield`/`golden_mirror` first — only valid from the round's non-starter; only in `playing` phase; in tournament the queue may contain up to 2 `swap` actions; the soldier's `special` carries a target `cell:{x,y}`, the werewolf's `special` a `dir` from 8 directions; countess/onre: `attack` carries a **diagonal** `dir`, `dash` is their **charge** (same 4 orthogonal dirs), `melee` has no dir (own cell only), `special` a trap `cell:{x,y}`), `retry`, `admin_reset_all`.
Server → client: `you_are` (`{ slot, isHost }`; slot is rolled at match start and fixed for the series), `color_roll` (after `configure_match` — trigger for the client-side colour-wheel animation; the seat assignment is already done), `state` (snapshot incl. `phase`/`config`/`series`, tournament also `roster`/`rosterReady`/`mageDead`/`rosterHp`/`rosterMana` + own `mageHp`/`mageMana`, optionally with `timeline`), `game_result` (per-game outcome), `new_game` (next game in series — reset round UI, keep score), `game_over` (series decided), `reset`.

### Game balance constants

All tuning lives at the top of `server.js`: HP/mana, costs, damage, and animation pacing (`*_MS` constants — currently 2× slowed). Client-side timing constants in `client.js` (`MOVE_MS`, `ATTACK_SWING_MS`, etc.) must stay in sync with the server's `delayMs` values or animations desync from the timeline playback.

Character specials differ by hit zone: fire hits the whole row (5 dmg) — and Fire is **no longer immune to Damage tiles** (takes the normal 1 dmg); instead, while he *stands on a Damage tile* when the special resolves, its range widens to his row **±1** (`fireOnDmgTile()` in `server.js`, mirrored in the client `cellsForSpecialPreview` fire branch reading `state.tiles`; from the middle row that is the whole 3-tall board, from an edge row 2 rows) — lightning hits every cell of the **opposite chess-color** to the one it stands on — half the board, and each orthogonal move flips its parity (3 dmg) — wanderer hits diagonal-adjacent only (8 dmg). Medusa's special is directional (left/right), deals no damage and **petrifies** (target skips its next 2 basic actions; golden actions are skipped without consuming the counter). Minotaur's special covers the **whole board**, deals no damage and banishes the opponent into the **labyrinth** (see below). Naruto's special is **self-range** and summons a **shadow clone** (see below). Escanor's special is **directional (left/right)** and deals **8 dmg** to a zone whose size grows with his **Pride level** (see below). The Soldier's special is a **player-picked target cell** — a sniper beam dealing **10 dmg** to whoever stands on that cell when it fires (see below). The Werewolf's special is an **8-directional charge** (diagonals included) that moves the caster and deals **2/4/6/8 dmg by his Moon phase** (see below). Vampire's and Onryō's special is a **player-picked hidden trap cell** (see below). The hit logic is in `specialDamageAndHit()` / the medusa+minotaur+naruto+escanor+soldier+werewolf+countess/onre branches of `doSpecial()` (server) and must match `cellsForSpecialPreview()` (client hover/cast preview) — they are maintained in parallel (the soldier's and the trap duo's "zone" travels in the action's `cell`, so their branches and the client picker/hover handle it instead of `specialZoneHas`; the werewolf's path depends on the opponent's live position — the client preview receives it via `meState.foe`).

**Pride (Escanor):** Escanor carries a per-mage **`pride` level 0–3** (`newPlayer`/`cloneActor`, public in snapshots). His directional special (`escanorCells(actor, dir)` ↔ client `cellsForSpecialPreview`) hits a zone anchored at the cell one step in the chosen direction `F`: **0** = just `F`; **1** = `F` + its 4 diagonals; **2** = the 3×3 around `F` (incl. Escanor's own cell); **3** = the whole board. **Throw off the board:** if `F` is out of bounds (cast toward the adjacent edge) the zone is empty **even at pride 3** — the whole throw goes out: the action still *executes* per the wall rule (mana spent, not crossed out), the choreography plays with the sun flying off the board and fading (no burst) and an OUT OF BOUNDS float; the zone previews (hover/ghost/HUD minimap) highlight nothing, same as aiming a basic attack off the board. Always 8 dmg, 5 mana, goes through defenses like any dmg special (shield blocks, mirror reflects 8 back). Pride evolves at **end of each round** in `resolveTurn`: if he queued **shield/mirror/golden shield/golden mirror** that round **or took a hit from the opponent** → **−1**, else **+1** (always a single ±1 step, the conditions don't stack; clamp 0–3); reset to 0 on character choice and on tournament swap-in. "Took a hit" = any HP loss caused by the opponent's action — incl. a mirror reflect of Escanor's own attack — tracked by `notePrideHit()` at every opponent-caused HP-deduction site (`applyHit`, `applyStackedHit`, the reflect branches of `applyHitOnClone`/`applyHitPairDefended`/`applyHitBoth`) into the per-round `player.prideHit` flag; tile/IK damage and the no-damage statuses (petrify, banish) do **not** count. **Pride is shown as the character's on-board size** — the board sprite scales `1.2^pride` from the feet (`escPrideMul`, `transform-origin` at `ESC_FEET_FRAC`; the YOU-flag, ghost and pair-shift math account for it) — a public tell the opponent reads. **Transform intro:** in-game Escanor starts as `WeakIdle` (weak "day" form); at the **first round's playback** (once per deployment — game start or swap-in, tracked by `escTransformed`) a one-shot `Transform` animation plays before any action (parallel if both are Escanor), after which the idle is `Idle` (Stand, strong form). **Special choreography:** the big centre cast sprite is `WinSun` (Escanor hoisting a growing sun) while the small board figure plays `WinSunBoard` — the raise plays **once**, then only the mouth moves and the sun flickers (the sheet's last 4 frames, looped via `ANIM_DEF` `loopFrom`, which needs a draw time relative to the anim start — `animState.start`); after the cast the small figure switches to `CruelSunHold`, during the sun's flight to the target it plays `WinTalk` (mouth only — no sun, no re-raise) and a `SunBurst`→`SunFade` FX (`spawnSunFxAt`) plays on the target cell. Assets live in `public/assets/escanor/` (+ `escanor_2/` = green clothing recoloured red for p2); the reproducible extraction pipeline is `tools/sprites-escanor/`.

**Shadow clone (Naruto):** the special (5 mana, cannot be blocked/mirrored) requires Naruto to stand on his cell **without the opponent** — the opponent standing on Naruto's cell is the **only way to block the summon**, so it still **costs the mana** (any existing clone is killed first — as in a recast — then seals play + a `not_alone` whiff, but no new clone spawns); a recast first kills the active clone (as if hit), then summons a fresh one on Naruto's cell. State lives in `player.clone = {x,y} | null` (serialized in snapshots; `redactActor` strips it for a cursed viewer). The clone **copies every basic action in parallel** with its owner — same frames/beats — with one inversion: **the vertical axis is mirrored for both movement and the basic attack** (owner moves up → clone moves down; owner shoots up → clone shoots down; horizontal is identical; dash keeps its own board clamp via `moveCloneSteps`). A vertical move/dash/attack at the board edge makes whichever figure is blocked play a **`wall_bump`** (OUT OF BOUNDS twitch) while the other still moves and mana is still spent; when *neither* figure can move the action still **executes** (both play `wall_bump`, mana still spent — it is never crossed out, see the "action into a wall" rule below). Because both figures split symmetrically on a vertical action, the opponent's **action log / round-script anonymizes Naruto's vertical direction to `↕`** (`displayDir` — up/down would reveal which figure is real; horizontal stays). Copied attacks/melee deal the **same damage as Naruto himself** — the basic shot uses full distance falloff (`Math.max(1, BASIC_DMG_MAX - dist)`) measured from the clone's own cell, melee deals the full `MELEE_DMG`, both scaled by the owner's Last Stand/Last Hope (`dealMul` ×2 / ×4) and maze (`labyrinthMul` ×2) multipliers. Because a clone **stacked on the owner's cell** fires/melees the same target in parallel, the pair lands two full hits = **double damage** (this stacks only for horizontal shots and melee; a vertical shot still splits up/down via the inversion). On an **undefended** target the two simultaneous hits merge into **one strike** (`applyStackedHit` → single `hit` frame with `dmg` total + `parts`; the client prints one „-n -n HP" float and HP drops at once), and the two identical horizontal projectiles render as a **single projectile** (the client dedupes the clone's charge that exactly overlaps the owner's). `CLONE_DMG = 1` now denotes only the clone's **soak** — the flat amount it eats when a shot passes through the stacked pair before the rest carries on to Naruto (its offense and its mirror reflect are both full, Naruto-equal); recharge is visual-only on the clone (mana is the owner's single pool — the `recharge` effect carries both cells so the client shows the aura on both); defenses are the **owner's shared `shield`/`mirror` flags** (armed and consumed together, identical glow on both sprites — the pair is visually indistinguishable; the client duplicates defense/mana floats via `cloneFloat`). The clone has no HP: **any connecting hit removes it** (`killClone` → `clone_die` frame) — enemy projectiles (a clone standing **apart** from the owner is a decoy that eats a shot whole and lets it fly on as smoke; a clone **stacked on the owner's cell** soaks only `CLONE_DMG` and the **rest of the shot passes through to the owner** — it is not a full shield, and in a labyrinth that surplus hit on the real owner reveals & ends the maze like any other), melee (**same stacked rule as a shot**: a decoy clone eats the strike whole; a **stacked** clone soaks only `CLONE_DMG` and the rest of the melee lands on the owner — undefended via `applyHitOnClone` + `applyHit`, a defended pair via `applyHitPairDefended`, and the stacked hit reveals & ends a labyrinth like a real hit), zone specials (they strike player **and** clone in the same action; a shared defense reacts on **both figures at once, each from its own cell, in a single synchronized beat** — the zone special and the demon route through `applyHitBoth(ownerSlot, ownerDmg, tl, kind, includeClone)`, which emits both blocks / both mirror-beams / both hits in the *same* frame so nothing plays sequentially: a **shield** breaks on the player *and* the clone (no damage either way), and a **mirror** reflects from both — both bounce the **full incoming dmg** (the clone reflects the same damage it took, exactly like the player's mirror — not a flat `CLONE_DMG`), so a mirrored wanderer special that hit both deals the attacker `8 + 8` **as one combined `hit` frame with `parts`** — a single „-8 -8 HP" float, HP deducted at once; undefended, a clone **stacked on the owner's cell** soaks `CLONE_DMG` off the zone/demon dmg (the owner takes the rest — same rule as a shot through the pair) while a decoy clone hit on its **own** cell dampens nothing (the owner takes the full dmg) — either way the clone poofs and the owner's hit land in the same beat. A **defended stacked pair** hit by a shot goes through `applyHitPairDefended` — the shared defense reacts as one figure in one beat: a single block frame, or a single mirror beam + one combined `parts` hit on the attacker (`fromClone` keeps the old semantics — reflect kills the attacker's clone and only one part lands as HP). `applyHitOnClone` (used for clone-only hits and the **undefended** stacked partial-absorb path, which stays sequential) reflects the **full incoming dmg** back at the attacker), Medusa's petrify (destroys the clone unless defended), the demon attack, and **IK tiles**. Minotaur's banish does **not** kill the clone — the labyrinth curse affects only the real owner, who keeps his clone in the maze; for both statuses a clone-side **mirror** still reflects the status onto the caster on clone-only contact. A dmg tile shows a cosmetic `clone_hit` −1 float but does not kill it; the clone never consumes heal/mana pickups (an untaken pickup betrays which figure is the clone). An opponent's mirror reflecting a **clone-originated** hit destroys the clone instead of damaging the owner (`applyHit(..., fromClone)`). The clone dies with the owner's death, swap (`doSwap`) and at game end (`resolveTurn` `ended` cleanup). **Clone × labyrinth:** a hit on a **decoy** clone (standing apart from the owner) never reveals nor ends a labyrinth and the cursed shooter must not learn about it — the killing projectile flies on to the board edge (smoke pass-through); a hit on a **stacked** clone is different, since the surplus damage lands on the real owner and that hit reveals & ends the maze normally and `clone_die`/`clone_hit` effects are redacted like other opponent-side effects; a hit **by** the clone is a normal action hit (ends the labyrinth, blocked/mirrored included). The clone counts for Ariadne-thread crossings both ways (`trackCloneSteps` + the clone check in `trackSteps` — the cursed player sees a silhouette at the meeting cell, not knowing it was the clone). Client-side the clone renders via per-slot `cloneEls` canvases (positioned in `positionActors`, drawn in the raf mirroring the owner's `animState` except hurt/dead — a tile hit on the clone triggers an independent `cloneFlinch` hurt-pose + `-N HP` float so it doesn't read as invulnerable; it does **not** mirror Last Stand visuals — `actorFilter(slot, now, true)` skips the golden death-glow. The clone is **always visible** (both sides — you can always see Naruto has a clone): while sharing the owner's cell it stands a few px beside him (`STACK_OFFSET`) and semi-transparent, and when separated it is full-opacity and identical; it **always slides** (never pops/vanishes in place), so splitting and merging read as two independent figures moving in parallel from the shared cell) plus a `cloneGhost` planning preview (`simulatedClonePos`); the summon choreography is `clone_summon` (owner + one copy on the same cell, mirrored to face the owner, both play `Special_2.png` via `spawnCloneSummonFx`) followed by `clone_born`.

**Sniper beam (Soldier):** the special (5 mana, **10 dmg**) carries a **target cell** picked on a mini-map that pops under the special button (`buildCellPicker` in `client.js`): the player may pick **any cell except the opponent's current cell** (and the cell of the opponent's shadow clone — both figures are blocked, otherwise the picker would leak which one is real) **and except his own ghost position** at that point of the queue (after his already-queued move/dash — `ghostPos()`; the server re-validates all of this in `validQueue()` with its own queue simulation incl. tournament swaps, and `randomSoldierTarget()` mirrors the rules for the timeout auto-fill). The point: the opponent takes the hit **only if they move onto the picked cell** during the round. The beam goes through defenses like any dmg special (shield blocks, mirror reflects the full 10) and never harms the Soldier himself (his own cell can't be picked; the beam only strikes the opponent/clone). A decoy clone that steps onto the cell dies to the beam via `applyHitOnClone` (no owner damage). **Soldier × labyrinth:** while the Soldier is cursed the opponent's cell is **not blocked** in the picker/validation (blocking it would reveal the hunter's position — the server redacts `x/y`, so the client couldn't block it anyway) — the cursed Soldier fires blind and **may hit a standing hunter**; a landing hit plays the usual `revealLabyrinths` → hit → `endLabyrinths` sequence. **Choreography:** one powerful sniper shot, not a burst — a single long **aim** frame (`SOLDIER_AIM_MS`, replaces the old 3× cast beats): the centre overlay and board figure **hold Shot_2's frame 0** (aiming pose, facing the target) while a faint red **laser sight** settles on the target cell (`spawnSoldierAim` + pulsing `.soldier-aim-dot`; the laser sway/lock-on is the CSS `soldier-aim` animation whose 2.3s must equal `SOLDIER_AIM_MS − SOLDIER_FIRE_MS`); in the last `SOLDIER_FIRE_MS` the fire frames 1–3 play **once** with a muzzle-flash brightness spike (`.soldier-fire`), then one `soldier_beam` frame (`SOLDIER_BEAM_MS`, must match client `spawnSoldierBeam`) — a thin red beam from the gun to the cell + an `Explosion.png` FX (`spawnExplosionAt`) — and only then the hit/block/mirror lands. Both the laser sight and the beam originate at the **actual muzzle tip** (`soldierGunOrigin` — sprite pixel measured from Shot_2's fire frame, mapped through the `positionActors`/`drawSprite` geometry; horizontal offset flips with shot direction, zero for purely vertical targets). `SOLDIER_AIM_MS`/`SOLDIER_FIRE_MS`/`SOLDIER_FIRE_FPS` exist in both `server.js` and `client.js` (aim) / `client.js` (fire) and must stay in sync. **Other actions:** basic attack throws a **grenade** (`Attack_1.png`→`Grenade.png` alias; the `Charge.png` projectile is a rolling grenade at ground level that **explodes on the last cell of its flight** — on the hit target or at the board edge, handled in `retireProjectile`), melee is a point-blank shot (`Attack_2.png`→`Shot_1.png`), and recharge plays his own `Recharge.png` pose (the only character with one). **Moon charge (Werewolf):** the werewolf carries a per-mage **`moon` phase 0–3** (new moon → crescent → half → full; `newPlayer`/`cloneActor`, public — the on-board size is a tell). **Blood trap (Vampire & Onryō):** two **side-bound** characters on the hidden char-select page (the corner-rune easter egg): `SIDE_CHARS = { countess: "p1", onre: "p2" }` — internal keys/asset folders keep the old names (`countess`/`Countess_Vampire`, `onre`/`Onre`), the display names are **Vampire** (p1 only) and **Onryō** (p2 only, referred to as **she/her** in all texts); they are **not in `CHARS`**, so they can't be drafted in a tournament (`choose_character` allows them only outside tournament, and `runeActive()` blocks the hidden page in roster- *and* team-mode). Onryō always plays on p2 in her **native palette** (`dirP2` points to the same `Onre/` folder → no albino filter). Both share one kit; the only difference is the **bonus** on a landed melee/charge/trap hit: Vampire **heals herself** (+`VAMP_MELEE_BONUS` 4 / `VAMP_CHARGE_BONUS` 2 / `VAMP_TRAP_BONUS` 3 HP), Onryō **drains the same amount of the foe's mana and gains it** (drain capped by what the foe actually has; own gain capped at 10). The bonus lands **only when the damage actually lands** — a block (incl. the mirror-immunity block) or a killed decoy clone yields nothing; UI texts describe the kit **without concrete numbers** (only the cost badges carry values). **Mirror immunity:** the opponent's mirror behaves like a **plain shield** against ALL their damage (`mirrorImmune()` branches in `applyHit`/`applyHitOnClone`/`applyHitPairDefended`/`applyHitBoth` emit a block frame instead of a reflect; the defense is consumed normally, and the blocked hit still ends a labyrinth). **Basic attack is diagonal with a wall bounce** — 4 diagonal directions (`DIAG_DIRS`; picker `#aim-picker-diag`): the shot **bounces once off any wall** (billiard — a side-wall bounce sends it back toward the caster), a **corner does not bounce** (the shot ends there; firing into a corner from a corner cell = wall-rule whiff), the **second wall contact ends the flight**, and the range is hard-capped at `VAMP_SHOT_RANGE` 3 cells → dmg exactly **3/2/1 by distance flown**. A shot into an adjacent wall bounces **immediately** (consuming the single bounce). The path lives in `vampShotRoute()` — **mirrored server↔client** (server flight/pre-scan + client `cellsForAimPreview`); each `charge` effect step carries the current `dir` so the projectile's **tip rotates into the flight direction after the bounce** (`spawnOrMoveProjectile` re-applies the orientation every step). **Melee (4 mana, `VAMP_MELEE_DMG` 8 + bonus 4)** strikes **only the caster's own cell** — no direction, no picker (empty = whiff, mana spent). **Dash is replaced by the Charge** (`doVampCharge`, same `dash` action slot/cost 4 mana, `VAMP_CHARGE_DMG` 4 + bonus 2): a run in one of 4 orthogonal directions that stops at the **first opposing figure** (decoy clone eats it whole) or the **board edge** (no target = pure repositioning); stays on the target's cell even when blocked; charge into the wall from an edge cell = wall rule. Both melee and charge may be queued in the same round (two action types). The charge moves the caster → modeled in the client ghost (`simulatedPositions`) and the server `validQueue` sim; the dash button gets the **purple flickering "Hidden page" skin with a ❔ icon** (`syncDashBtn` + `.btn.act.dash.vamp-charge`). **Special (5 mana) = hidden trap** (`player.trap = {x,y} | null`): a cell picked on the soldier-style mini-map with **no restrictions** (even the foe's current cell — it triggers only on a *re-entry*, standing doesn't trigger; even the caster's own cell). The marker (a static head cut from `Idle` frame 0, `renderTrapMarkers` + `HEAD_CROP`) is visible **only to the caster** — the server strips `trap` from the opponent's view of the actor and filters the `trap_set` effect in **every** snapshot & timeline frame (`snapshotFor` + `maskTrapFrame` in `redactTimelineFor`); the cast blinks the **whole board** like the Minotaur's — a full-board blink leaks nothing, the target cell itself is never highlighted (Countess's cast is a **single long `VAMP_CAST_MS` frame** — the big centre sprite and the board figure both play the whole **`A5.png`** montage once from frame 0, one-shot via `dataset.once`/relative `drawT`; Onre keeps the classic `SPECIAL_REPEAT` beats with his looping `Scream`). Max 1 trap; recasting replaces it. **Trigger:** the foe (or their shadow clone) **entering or crossing** the cell during an action — evaluated **after the foe's action fully completes** (a per-action visited-cells collector `actionSteps` is filled by `trackSteps`/`trackCloneSteps` and drained by `resolveTrapsAfterAction` in `resolveTurn`; a werewolf charge finishes its whole strike first). The trigger always **consumes the trap** and always **teleports the caster** to the cell — even without mana for the strike and even when the foe merely sprinted through (`trap_trigger` flare — from that moment both players see where it was — then `trap_tp_out`/`trap_tp_in`, thread cells counted); the melee (3 dmg + bonus, **another 4 mana**) fires only if the caster can afford it (otherwise a `trap_no_mana` float — teleport only). **Melee/trigger choreography** (per `vampire_attack_guide.png`, rebuilt 1:1 as the **A-sheets `A1`–`A5.png`** with the `Blood_Charge` effects baked into the frames — generated by `tools/sprites-countess-onre/build.cjs`, no separate blood-FX overlays anymore): Vampire's strike is **`A1`** (anim `va1`) **unified across all three melee forms** — the own-cell melee (one `vamp_strike` frame instead of the generic 3 swing beats), the dash-slot charge and the trap trigger — the damage lands only **after A1 completes**, and on a landed hit she then plays the **`A2` → `A3` → `A4`** heal beats (`vamp_heal_cast` frames pushed by `vampBonus`, `VAMP_HEAL_BEAT_MS`) with her heal arriving only **after them** (never simultaneously with the hit); A1 and the heal beats mirror into the **big centre sprite** as one-shots (`meleeCasts` with `once` → `dataset.born`, replayed from frame 0 per beat). On a block/killed decoy/empty whiff only A1 plays — no heal beats. Onryō keeps her original choreography everywhere: at the trap she appears with a **`Scream`** (played on the `trap_tp_in` frame; Vampire appears in idle), strikes with `Attack_1` (`vattack1`; charge strike = `VampStrike` alias, own-cell melee = the generic 3 beats) and her drain lands right after the hit. A foe who merely **ran through** eats a strike into an empty cell — the 4 mana are spent (the intended way to bait/counter the trap). The trap melee goes through defenses normally and consumes the struck figure's armed defense; a whiff consumes nothing. In a **Countess-vs-Onre** match both sides can have traps and a trigger teleport can chain-trigger the other side's trap (finite — each trap is consumed). **Own entry destroys the own trap**; a **petrified (or dead) caster** doesn't trigger — the foe's crossing **destroys** the trap instead; the Last Stand ritual death destroys it too — all via `trap_break`, whose destruction FX (`spawnTrapBreak`) is deliberately **visible to both** players (the foe learns the risk is gone). The trap survives rounds and dies on caster death, swap and game end. **Trap × labyrinth:** the trap works while a labyrinth runs; teleporting onto the hunter's crossing cell is an intended positional leak. A trap melee that actually strikes the foe plays the standard `revealLabyrinths` → hit → `endLabyrinths` sequence; a whiff (or a killed decoy) leaves the maze running — and for a cursed viewer all opponent-side trap effects are redacted as usual. **Assets:** `public/assets/Countess_Vampire/` (strike = `A1.png`, heal beats = `A2`–`A4.png`, special cast = `A5.png` — the A-sheets are 128×128 strips composing `Attack_1–4` with the `Blood_Charge_1–4` effects per the attack guide; note the Blood_Charge sheets have **64×48 frames**, not 48×48) and `public/assets/Onre/` (strike = `Attack_3.png` via the fictitious `VampStrike.png`, trap strike = `Attack_1.png`, cast = `Scream.png`). Both `Charge.png` projectiles are **generated**: the same bolt shape re-centred into square **48×48 frames** (the raw 64×48 frames would be mis-sliced by the width/height frame inference and jitter in flight) — red for Vampire, ghostly-blue for Onryō. The reproducible pipeline (Charge generation, A1–A5 composition, head bbox measurement) is `tools/sprites-countess-onre/build.cjs`.

**Labyrinth (Minotaur):** the cursed player sees only the cell they stand on **plus Damage tiles and all IKs** (`redactTilesFor`: dmg tiles + `iks` are always visible so the blind player can dodge them; **heal/mana tiles are redacted unless the cursed player — or their own shadow clone — stands directly on that cell**) — the server **redacts their data** (`snapshotFor` + `redactTimelineFor` in `emitStateMasked()`): opponent position/defenses/**mana** and all opponent-originated timeline effects are stripped (incl. `tile_proc` blinks outside the player's own cell); opponent `action` announcements are masked as `{type:"unknown", beat}` so the round-script bar shows "?" on the right beat. Redaction is **two-way**: the hunter (Minotaur) does *not* see the Ariadne's thread, the `threadMark`, `thread_mark` effects, nor the cursed player's **mana** (`redactHunterActor`) — during the labyrinth neither side can track the opponent's mana (HP stays in the data but is visually hidden too); on **both** sides the opponent's HUD widget is **covered** by `assets/minotaur/labyrint.png` (`.hud-box.in-maze .maze-cover` — portrait, HP, mana, action log and the YOU/OPPONENT tag are hidden; the starter flag and series crowns stay visible). While anyone is in the labyrinth the arena background dims for **both** players (`body.labyrinth-mode`); the cell fog applies only to the cursed one. The effect lasts until **either player lands an action hit on the other — a blocked or mirrored hit counts too** (`endLabyrinths()` is called in `applyHit`/`applyPetrify`/`applyLabyrinth` *after* the block/reflect/hit frames, so the hit animation and HP loss play out before the ESCAPED float and the arena un-dims; in `applyLabyrinth` it runs before the new curse lands so two labyrinths never run at once; tile damage does *not* end it). Exception — **deterministic deaths during a labyrinth**: a lethal tile/IK hit and the Last Stand doom banish play the full reveal sequence (`revealLabyrinths` before their frames, `endLabyrinths` after the death) so a game never ends inside the fog with redaction still active. Because a hit is deterministic at action start, an action that will land one triggers `revealLabyrinths()` **before its animation frames**: a `labyrinth_reveal` frame sets the actor flag `labReveal`, which drops the redaction, the fog and the widget cover immediately (the hidden opponent fades in client-side, demon-style; thread and `threadMark` die at reveal) while `labyrinth` itself stays true through the action — the arena stays dimmed until the real `labyrinth_end` lands after the hit. While cursed, every cell the player enters (dash counts intermediate cells) extends their **Ariadne's thread** (`player.thread`, starts on the cell where they were cursed); `player.threadMark` records the **last cell where the thread and the Minotaur met** — whether the Minotaur stepped onto the thread *or* the cursed player walked onto the Minotaur's cell — and the cursed player sees the opponent's silhouette there. **Hunter on a torch-lit cell:** the cursed player's **torch cells** — their own cell plus the cell of their own shadow clone (the clone is a second torch: it lights its cell identically, incl. revealing heal/mana tiles under it, but never weaves the thread) — are always visible ("torch-lit", `.cell.torch-lit`). When a hunter figure (the hunter or its shadow clone) currently stands on a torch cell the server exposes `hunterHere` on the cursed player's own snapshot actor — a **list of the hunter figures' cells** (`hunterLitCells` in `redactSnapshotFor`/`redactTimelineFor`, max 2 entries, a duplicate cell = the hunter's clone stacked on its owner; the only positional info leaked, recomputed each snapshot/frame so it clears the instant the hunter leaves; the hunter's real `x/y` stay redacted). Both figures must behave identically since the hunter can't tell which is the real one. The client then renders each such figure **lit** (real palette, not the black `brightness(0)` shadow — `.sprite-silhouette.lit`, two pooled canvases `actorSilhouettes`) exactly as the hunter sees that tile: normal shared-cell behaviour — char-aware `pairShift` step-aside **on both sides** (`pairShift`/the clone's `foeShift` have a `hunterHere` branch so the cursed player's own figure steps aside too), facing each other, live idle loop, Escanor/werewolf level size applied; without a lit figure the silhouette stays the black static `threadMark` breadcrumb. Thread and mark die with the labyrinth. The special goes through defenses like petrify: shield blocks it, mirror reflects it — the Minotaur himself ends up in the labyrinth. Casting it on an already-cursed target is invalid (`already_lost`). **Balance — maze buff (`player.mazeBuff`):** a **successful direct cast** (curse actually lands on the foe — not blocked, not mirror-reflected) grants the caster `mazeBuff` for the labyrinth's duration (set in `applyLabyrinth`'s direct branch, cleared in `endLabyrinths`). It gives the caster **2× outgoing damage** (`labyrinthMul` multiplies the raw in `doBasic`/`doMelee` — since it's *outgoing*, a mirror reflects the **doubled** damage back onto the Minotaur); the caster takes **Damage & IK tiles normally** (as if outside the labyrinth — `endOfStepTileEffects` no longer skips them). It is granted only to the player who *cast* it, **regardless of character** — a mirror-created labyrinth grants **no** buff to anyone (the mirrorer never cast it; the reflected Minotaur is the cursed one). This makes the buff exclusive to the player actually playing the hunter Minotaur. Client-side the fog/thread/silhouette are rendered in `renderGrid()` (cells `.fogged`/`.thread-cell`, `renderThreadLines()`) and the raf (silhouette canvas + `.in-maze` widget cover); a hidden opponent has `x: null` — rendering paths guard on it (incl. `pairShift`, so a shared cell never leaks to the cursed player).

Defenses cover **the opponent's next action** after activation and are consumed by it even if it dealt no damage: shield (2 mana) blocks all damage, mirror (4 mana) reflects the full damage back at the attacker (applied raw — reflected damage cannot be shielded or re-reflected), golden shield (3 mana, pre-round extra action of the non-starter) sets the same `shield` flag, and golden mirror (5 mana, the same pre-round slot) sets the same `mirror` flag. The golden pre-action variants carry a `shieldGold`/`mirrorGold` flag purely for gold-tinted visuals (the `block`/`mirror` timeline effects carry a `gold` boolean). All are armed/consumed in the `resolveTurn()` loop, applied in `applyHit()`, and expire at end of round — they do **not** carry over. Strict interleaving means two own defenses are never armed simultaneously — each one covers exactly the opponent action that follows its activation.

Special tiles spawn at the end of every round — **`config.tilesPerRound` (1–3) rolls per round**, each typed by `config.tileWeights` (`dmg`/`heal`/`mana`/`ik` percentages summing to 100; `rollTileType()`). They resolve at the **end of each step** in `endOfStepTileEffects()`: dmg tiles are permanent (1 dmg if standing on them), heal/mana are the only consumables (taken by the round starter if both players share the cell; a heal tile gives +1 HP — except **Lightning**, whom it heals to full HP), and IK tiles (10 dmg) are overlays that hide tiles beneath them. **IK is now a list (`game.iks`), multiple allowed** — every IK relocates each round (`relocateIKs()`) and may overlap other tiles and players; the only constraint is that two IK tiles never share a cell. Tile damage bypasses all defenses (shield/mirror).

HP and mana both cap at 10 and are rendered in the HUD as 10-segment bars (`renderBar()` in `client.js`). Tile size is rectangular, driven by the `--tile-w`/`--tile-h` CSS variables, which `client.js` reads at startup (`TILE_W`/`TILE_H`).

### Sprites and assets

Sprite sheets are horizontal strips in `public/assets/<char>/`; frame count is inferred from `width / height` (frames must be square). Animation definitions (file, fps, loop) live in `ANIM_DEF` / `SPECIAL_ANIMS` in `client.js`. Characters with a native right-side palette (medusa, minotaur, naruto) have `dirP2` in `CHAR_META` (two sprite folders) instead of the CSS `alt-color` filter; medusa/minotaur `Charge.png` projectiles are per-palette recolored copies of fire's fireball, naruto's is a chakra spiral (P2 gold). Minotaur's sheet only has Idle/Walk/Attack/Hurt/Dead — `SPRITE_FILE_ALIAS` in `client.js` maps the expected `Run.png`/`Attack_1.png`/`Attack_2.png` names onto them. Naruto's assets were extracted from a ripped sheet via the reproducible pipeline in `tools/sprites-naruto/` (see its README); `Special.png` = hand seals (special cast), `Special_2.png` = the clone-summon pose loop. Arena background layers are listed in the client-side `ARENAS_CLIENT` map and loaded from `public/arenas/<key>/`. Background layers render into low-res canvases (`ARENA_RES`, upscaled with `image-rendering: pixelated`).

### Admin reset

`GET /admin/reset-all?key=…` or the socket event `admin_reset_all` disconnects both players and recreates the game. The client shows a reset button when loaded with `?admin=1` (optionally `&key=…`).
